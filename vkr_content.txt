ВВЕДЕНИЕ
В условиях непрерывного развития информационных технологий профессия инженера-программиста остается одной из самых востребованных на мировом рынке труда. Язык программирования Java на протяжении многих лет является индустриальным стандартом для разработки распределенных высоконагруженных систем (Enterprise), микросервисной архитектуры и мобильных приложений. Изучение Java требует глубокого понимания принципов объектно-ориентированного программирования, статической типизации, алгоритмов и структур данных, а также механизмов управления памятью (Garbage Collection).
Традиционные методы обучения программированию, опирающиеся на чтение лекций и выполнение лабораторных работ в локальной среде разработки (IDE), обладают существенным недостатком: они требуют от студента самостоятельной настройки окружения, что часто вызывает затруднения на начальных этапах, и не обеспечивают немедленной объективной оценки правильности написанного кода. Решением данной проблемы является создание специализированных обучающих веб-платформ с интегрированной системой автоматической проверки кода (Judge System) и элементами геймификации.
Разработка надежной и безопасной серверной части (backend) для такой платформы представляет собой нетривиальную инженерную задачу. Сервер должен не только обеспечивать стандартный функционал (аутентификация, работа с базой данных, предоставление REST API), но и предоставлять изолированную среду (Sandbox) для компиляции и выполнения потенциально опасного пользовательского кода. Любая уязвимость в механизме изоляции может привести к компрометации всей системы (выполнение произвольного кода на сервере, исчерпание вычислительных ресурсов, утечка данных). Кроме того, серверная часть должна эффективно взаимодействовать с клиентским игровым приложением, разработанным на движке Unity, обеспечивая реал-тайм передачу метрик прогресса и синхронизацию состояния.
Объект исследования: архитектура высоконагруженных серверных систем для образовательных платформ.
Предмет исследования: алгоритмы безопасного выполнения недоверенного программного кода (Java), модели данных нереляционных СУБД и протоколы интеграции веб-сервисов с игровыми движками (Unity WebGL).
Цель работы: разработка серверной инфраструктуры игрового веб-приложения для изучения языка Java, включающей защищенную подсистему компиляции кода и API для взаимодействия с Unity-клиентом.
Для достижения поставленной цели определены следующие задачи:
провести анализ существующих образовательных платформ и методов обеспечения безопасности при выполнении недоверенного кода,
спроектировать архитектуру серверного приложения, обеспечивающую асинхронную обработку запросов,
разработать оптимальную структуру документо-ориентированной базы данных для хранения профилей, достижений и образовательного контента,
выбрать технологии  для реализации…
разработать и настроить изолированную среду выполнения Java-кода,
разработать RESTful API и интегрировать технологию Server-Sent Events (SSE) для асинхронного уведомления клиента о результатах проверки кода.
спроектировать и реализовать модуль интеграции с игровым движком Unity,
провести нагрузочное тестирование сервера и тестирование безопасности Sandbox-среды.
Научная новизна: предложена оптимизированная модель распределения вычислительных ресурсов при массовой проверке Java-кода с использованием "горячего" пула Docker-контейнеров, интегрированная с геймифицированным API, который асинхронно синхронизирует прогресс обучения с Unity-клиентом посредством паттерна Server-Sent Events.
Практическая значимость: разработанная серверная инфраструктура является полностью независимым и защищенным модулем, который может быть использован не только в связке с текущим Unity-клиентом, но и как самостоятельное API для проведения олимпиад по программированию или интеграции в сторонние LMS-системы.
1 Теоретико-аналитические основы разработки серверной части образовательной игровой платформы
Анализ предметной области: образовательные платформы нового поколения
Рост цифровизации образования в последние десятилетия привёл к формированию принципиально нового класса программных систем — образовательных платформ с автоматической проверкой заданий. Несмотря на то, что первые онлайн-системы дистанционного обучения появились ещё в 1990-х годах в форме статических курсов на основе HTML, подлинный технологический прорыв произошёл в середине 2010-х, когда ведущие технологические компании осознали экономический потенциал массового дистанционного образования. Именно тогда началась разработка таких платформ, как Coursera (Стэнфордский университет, 2012), edX (Массачусетский технологический институт, 2012) и Udacity (2011). Однако первая волна таких платформ была ориентирована преимущественно на пассивное потребление контента: просмотр видеолекций, выполнение тестов с закрытыми вопросами.
Следующим шагом в эволюции образовательных платформ стало появление систем с интерактивной проверкой программного кода в реальном времени. Ключевой особенностью этих систем является наличие автоматизированного судьи (Judge System), который принимает написанный студентом исходный код, компилирует его в защищённой среде, запускает на заранее подготовленных наборах тестов и мгновенно возвращает результат. Это позволило сделать обучение программированию по-настоящему практическим, не требующим участия преподавателя на каждом шаге. Платформы типа LeetCode, Codeforces, HackerRank и отечественный Яндекс Контест стали стандартом для отработки алгоритмических навыков.
Тем не менее, несмотря на технологическую зрелость механизмов проверки кода, перечисленные платформы обладают существенным педагогическим недостатком: они ориентированы на уже подготовленного пользователя, знающего синтаксис языка, и не предоставляют ни образовательного нарратива, ни прогрессивной системы обучения «с нуля». Для начинающего студента открытый список из трёх тысяч задач без контекста является источником фрустрации, а не мотивации.
Параллельно с развитием алгоритмических платформ сформировался другой класс систем — геймифицированные образовательные платформы. Отличительной их чертой является акцент на вовлечённость и мотивацию через игровые механики: прогрессивное повышение сложности, систему наград и достижений, персонализированные профили с визуализацией прогресса и элементы соревновательности. Ярким примером является российский сервис JavaRush (выпущен под глобальным брендом CodeGym), в котором изучение Java подаётся в виде фантастического квеста с развёрнутым нарративом и тысячами практических мини-задач.
Таким образом, анализ существующих платформ выявил неудовлетворённую нишу: отсутствует система, которая сочетала бы в себе промышленное качество механизма автоматической проверки Java-кода (сравнимое с LeetCode), полноценную геймификацию с визуальным представлением в виде 3D/2D игры (как в CodeCombat) и современную веб-архитектуру с открытым исходным кодом, пригодную для масштабирования и адаптации.
1.2 Психологические основы геймификации в техническом образовании
Понятие «геймификация» (от английского gamification) было введено в научный оборот в 2002 году британским исследователем Ником Пеллингом, однако широкое распространение в академических кругах получило примерно в 2010–2012 годах. Под геймификацией понимается применение игровых элементов, механик и принципов мышления в неигровом контексте с целью повышения вовлечённости, мотивации и достижения измеримых результатов.
Теоретической основой геймификации служат несколько концепций из области психологии мотивации. Прежде всего, это теория самодетерминации (Self-Determination Theory, SDT), разработанная Эдвардом Деси и Ричардом Райаном в 1980-х годах. Согласно этой теории, устойчивая внутренняя мотивация человека определяется удовлетворением трёх базовых психологических потребностей: потребности в компетентности (ощущение роста и мастерства), потребности в автономии (чувство контроля над своими действиями) и потребности в социальной связанности (принадлежность к сообществу). Грамотно спроектированная система геймификации напрямую адресует все три потребности: система уровней и достижений удовлетворяет потребность в компетентности, возможность выбирать курсы и задачи — потребность в автономии, а таблицы лидеров и система друзей — потребность в социальной связанности.
Второй важной теоретической основой является концепция потока (Flow Theory) Михая Чиксентмихайи. Состояние потока — это особое психологическое состояние полного погружения в деятельность, характеризующееся потерей ощущения времени и высокой продуктивностью. Чиксентмихайи показал, что состояние потока достигается при балансе между уровнем сложности задачи и уровнем навыка исполнителя. Если задача слишком проста — наступает скука; если слишком сложна — возникает тревожность и фрустрация. В образовательных платформах с системой геймификации этот баланс поддерживается за счёт адаптивного нарастания сложности: каждая следующая задача чуть сложнее предыдущей, что постоянно удерживает студента в зоне потока.
Применительно к обучению программированию эти теории имеют особое значение. Изучение языка Java объективно является одним из наиболее когнитивно нагруженных видов деятельности для новичка: необходимо одновременно усваивать синтаксис, концепции объектно-ориентированного программирования, логику работы виртуальной машины и алгоритмическое мышление. Без правильной системы мотивации студенты часто прерывают обучение после первых неудач. Внедрение геймификации позволяет переосмыслить ошибку компиляции не как провал, а как обратную связь на пути к следующему уровню, что кардинально меняет эмоциональное восприятие процесса обучения.
1.3 Систематический сравнительный анализ существующих платформ
Для обоснования уникальности разрабатываемой системы был проведён систематический сравнительный анализ семи наиболее значимых образовательных платформ по ключевым критериям.
LeetCode — американская платформа, основанная в 2015 году. Специализируется на алгоритмических задачах для подготовки к техническим собеседованиям. Поддерживает более 25 языков программирования, включая Java. Автоматическая проверка кода осуществляется посредством закрытых тест-кейсов с измерением времени выполнения и объёма потреблённой памяти. Из игровых механик присутствует только глобальный рейтинговый соревновательный режим (Weekly Contest). Образовательного нарратива нет, платформа рассчитана на подготовленного пользователя.
HackerRank — платформа, активно используемая рекрутёрами IT-компаний для технического скрининга кандидатов. Помимо алгоритмических задач, предлагает тематические треки по конкретным технологиям. Геймификация представлена значками за выполнение наборов заданий и глобальным рейтингом. Практической системы обучения «с нуля» не предусмотрено.
Stepik — российская платформа, изначально созданная Петербургским университетом для создания онлайн-курсов. Поддерживает проверку Python-кода, наличие Java-тестирования ограничено. Основной формат — видеолекции с тестами. Геймификация сведена к минимуму: баллы за выполнение заданий и сертификаты. Отсутствует игровой компонент.
Codecademy — американская платформа с упором на интерактивные уроки в браузере. Пользователь видит задание, пишет код в встроенном редакторе, получает немедленный результат. Геймификация выражена в достижениях (Badges) и «сериях» выполненных дней (Streak). Игровая составляющая декларативная — нет реального визуального игрового процесса.
JavaRush / CodeGym — наиболее близкий аналог разрабатываемой системы среди существующих решений. Платформа выстроена вокруг фантастического нарратива: студент — это «робот», обучающийся у инопланетян. Имеется около четырёх тысяч задач, ориентированных именно на особенности Java. Проверка осуществляется через встроенный движок или плагин для IntelliJ IDEA. Недостатки: закрытый проприетарный код, высокая стоимость подписки, отсутствие настоящего 3D/2D игрового рендеринга (игровой элемент только нарративный и текстовый).
CodeCombat — американская платформа с реальным игровым рендерингом. Пользователь управляет персонажем на экране, напрямую вводя игровые команды. Это обеспечивает наиболее высокий уровень когнитивной связи между написанным кодом и визуальным результатом. Однако платформа ориентирована преимущественно на детей и подростков; Java поддерживается в ограниченном объёме; изучение продвинутых концепций языка (многопоточность, паттерны проектирования) невозможно.
CheckiO — платформа с элементами геймификации, где задачи подаются как миссии на отдельных островах. Поддерживает Python и TypeScript, но не Java.
Результаты сравнительного анализа подтверждают вывод, сформулированный в разделе 1.1: ни одна из существующих платформ не предоставляет одновременно промышленного уровня Java-проверки, визуального игрового рендеринга через современный 3D-движок и открытой масштабируемой серверной архитектуры на Node.js.
1.4 Исследование технологий автоматической проверки программного кода
Автоматическая проверка программного кода (Automated Code Judging) является центральным техническим компонентом любой платформы для обучения программированию. Историческое развитие этой технологии прошло несколько этапов.
На первом этапе (1970–1990-е) автоматические судьи использовались исключительно в контексте олимпиадного программирования. Системы типа PC^2 (Programming Contest Control) и ejudge использовали низкоуровневые механизмы операционной системы Linux для изоляции выполняемых процессов. Перехват системных вызовов осуществлялся через системный вызов ptrace(), что позволяло контролировать каждое обращение программы-студента к ядру ОС. Эти системы обеспечивали высокий уровень безопасности, но имели два существенных ограничения: во-первых, они требовали компиляции в нативный машинный код, что делало их неприменимыми для JVM-языков; во-вторых, перехват ptrace() создавал значительные накладные расходы на производительность.
На втором этапе (2000–2010-е) с ростом популярности JVM-языков (Java, Scala, Kotlin) возникла необходимость в новых подходах. Поскольку программы на Java исполняются не нативно, а внутри виртуальной машины JVM, стандартный механизм ptrace не мог контролировать их на должном уровне. В качестве альтернативы исследовались подходы на основе менеджеров безопасности (Java SecurityManager) — специального API самой JVM, позволяющего ограничивать права выполняемого кода изнутри. Однако SecurityManager имел принципиальный недостаток: он не мог ограничивать потребление процессорного времени и памяти, что делало систему уязвимой для DoS-атак.
На третьем этапе (с середины 2010-х по настоящее время) доминирующим подходом стала контейнеризация на базе Docker. Технология контейнеров Linux (LXC и Docker) позволила объединить преимущества обоих предыдущих подходов: строгое ограничение ресурсов через механизм Cgroups ядра Linux и изоляцию файловой системы через Namespaces, при этом не накладывая перехвата ptrace и не требуя модификации исполняемой JVM. Именно этот подход был принят в разрабатываемой системе.
1.5 Механизмы изоляции процессов на уровне ядра Linux
Для глубокого понимания безопасности разрабатываемого Backend-сервиса необходимо рассмотреть технологические механизмы, на которых строится Docker.
Пространства имён (Namespaces). Linux Namespaces — это механизм ядра, позволяющий создавать изолированные контексты выполнения. Каждый контейнер Docker работает в собственных пространствах имён нескольких типов. PID Namespace изолирует дерево процессов: процессы внутри контейнера видят только друг друга и не могут посылать сигналы (kill) процессам хост-системы. Mount Namespace изолирует файловую систему: контейнер видит только те каталоги, которые явно «примонтированы» (mounted) при его запуске, и не имеет доступа к /etc/passwd или /var/secrets хост-машины. Network Namespace изолирует сетевой стек: контейнер может иметь собственный виртуальный сетевой интерфейс или быть запущен вообще без сетевого доступа (параметр --network none), что полностью исключает возможность несанкционированного взаимодействия выполняемого кода с внешними сетевыми узлами.
Контрольные группы (Cgroups). Если Namespaces отвечают за изоляцию видимости ресурсов, то Cgroups версии 2 (cgroupv2), встроенные в современные ядра Linux (начиная с версии 4.x), отвечают за ограничение потребления ресурсов. Подсистема memory.max позволяет задать максимальный объём памяти (включая кучу JVM, стек потоков и кеш страниц), который разрешено использовать группе процессов. При превышении лимита ядро Linux немедленно завершает (kill) нарушителя с кодом 137 (SIGKILL), не давая ему возможности обработать ситуацию. Подсистема cpu.max ограничивает долю процессорного времени: например, значение «50000 100000» означает, что контейнер имеет право использовать не более 50 миллисекунд из каждых 100 миллисекунд реального времени — фактически не более 0.5 ядра CPU.
Наложенная файловая система (OverlayFS). Docker использует технологию Overlay Filesystem для эффективного хранения образов контейнеров. Образ состоит из иерархии слоёв, доступных только для чтения (read-only layers). При запуске контейнера поверх них накладывается тонкий слой чтения-записи (read-write layer), уникальный для каждого контейнера. Все изменения, которые производит выполняемый код (создание файлов, запись в /tmp), происходят только в этом верхнем слое и автоматически уничтожаются при завершении контейнера с флагом --rm. Это гарантирует полную очистку после каждой проверки.
1.6 Специфика языка Java применительно к системам автоматической проверки
Язык программирования Java обладает рядом архитектурных особенностей, отличающих его от компилируемых языков (C, C++, Go) и интерпретируемых скриптовых языков (Python, JavaScript). Эти особенности необходимо принимать во внимание при проектировании Backend-модуля проверки кода.
Принципиальное отличие Java заключается в двухэтапном процессе трансляции. На первом этапе компилятор javac преобразует исходный текст (.java файлы) в платформо-независимый байт-код (.class файлы). На втором этапе виртуальная машина Java (JVM) интерпретирует или JIT-компилирует байт-код в нативные инструкции конкретного процессора. Это означает, что для запуска программы студента на сервере необходимо выполнить две отдельные команды с разными порогами тайм-аута: компиляция (javac) и выполнение (java).
Критически важной особенностью является высокий базовый уровень потребления оперативной памяти процессом JVM. Даже простейшая программа при старте JVM резервирует значительный объём памяти под несколько пулов: память под кучу (Heap) для хранения объектов, под стек каждого потока (Thread Stack), под метаданные загруженных классов (Metaspace) и под внутренние структуры JVM. Если ограничить контейнер Docker чрезмерно жёстко, процесс JVM может завершиться с ошибкой инициализации ещё до того, как успеет выполнить код студента. Для предотвращения этого при запуске Java внутри контейнера используются специальные флаги JVM, явно регулирующие размеры пулов памяти.
Ещё одна важная особенность связана с многопоточностью. Современные JVM по умолчанию запускают несколько фоновых потоков: поток сборщика мусора (Garbage Collector Thread), поток финализации объектов, потоки внутренней компиляции JIT. При использовании сборщика мусора G1GC или ZGC количество вспомогательных потоков масштабируется пропорционально числу доступных процессорных ядер. Это означает, что ограничение контейнера до 0.5 CPU не обязательно означает ограничение в 0.5 CPU для кода студента — часть времени будет «съедена» инфраструктурой JVM. Для минимизации этого эффекта при разработке системы был выбран последовательный сборщик мусора (SerialGC), который является однопоточным и лучше подходит для краткоживущих изолированных процессов.
1.7 Технология Server-Sent Events (SSE) как механизм асинхронного взаимодействия
В классической модели взаимодействия HTTP-клиента и сервера инициатором обмена всегда выступает клиент. Клиент отправляет запрос — сервер формирует ответ и закрывает соединение. Такая модель отлично подходит для получения данных по запросу (профиль пользователя, список курсов), однако создаёт проблемы в сценариях, где сервер должен уведомить клиента об асинхронном событии (завершение длительного процесса компиляции, входящий запрос в друзья).
Существует несколько стандартных подходов к реализации серверных уведомлений. Техника Short Polling предполагает регулярную отправку запросов клиентом с заданным интервалом (например, раз в секунду). Простота реализации сопровождается высокой нагрузкой на сервер и задержкой уведомления, равной периоду опроса. Техника Long Polling удерживает HTTP-соединение открытым до тех пор, пока сервер не сформирует ответ, после чего клиент немедленно переустанавливает соединение. Это снижает среднюю задержку, однако технически реализация в Node.js требует аккуратного управления жизненным циклом объекта ответа (res) для предотвращения утечек памяти.
WebSocket — двунаправленный протокол с постоянным соединением, поддерживающий отправку сообщений как от клиента к серверу, так и от сервера к клиенту. Обеспечивает минимальную задержку, но требует специализированной реализации на стороне клиента и более сложной инфраструктуры масштабирования (при горизонтальном масштабировании серверных нод необходим отдельный брокер сообщений, например Redis Pub/Sub).
Server-Sent Events (SSE) — стандарт W3C, реализующий однонаправленный поток событий от сервера к клиенту поверх обычного HTTP. Сервер устанавливает заголовок Content-Type: text/event-stream и удерживает соединение открытым, периодически записывая в поток ответа строки в специфическом текстовом формате. Клиент (браузер или C#-код в Unity) читает этот поток построчно, реагируя на приходящие события. Преимущества SSE по сравнению с WebSocket: нативная поддержка в браузере без дополнительных библиотек, автоматическое переподключение при обрыве соединения, совместимость со стандартными HTTP-прокси и балансировщиками нагрузки. Именно технология SSE была выбрана в разрабатываемой системе для организации push-уведомлений из Node.js-сервера в Unity-клиент.
1.8 Обоснование выбора Firebase Realtime Database и Firestore
Для хранения данных в разрабатываемой системе используются два продукта платформы Google Firebase, каждый из которых оптимален для своего класса задач.
Firebase Realtime Database — одна из первых NoSQL баз данных Firebase, хранящая данные в виде единого дерева JSON. Её главным достоинством является мгновенная синхронизация данных между клиентами в реальном времени через WebSocket. В разрабатываемой системе Realtime Database используется для хранения двух категорий данных, требующих высокой скорости записи: данных системы достижений (коллекция achievements и userAchievements) и данных системы друзей (коллекция friends). Архитектура друзей хранит связи в виде вложенных узлов: для каждого пользователя хранится словарь (dictionary), где ключами являются UID других пользователей, а значениями — объект со статусом отношения (pending_sent, pending_received, active) и временной меткой. Это позволяет за одно чтение получить весь список друзей пользователя.
Cloud Firestore — более новый и масштабируемый продукт Firebase, организованный по документо-ориентированной модели. В отличие от Realtime Database, Firestore поддерживает сложные составные запросы (composite queries) с фильтрацией и сортировкой по нескольким полям, что критически важно для реализации функции глобального рейтинга (сортировка пользователей по убыванию поля xp). Firestore используется для хранения профилей пользователей, образовательного контента (курсов и уроков) и истории попыток (submissions).
Разделение данных между двумя базами позволяет использовать преимущества обеих систем: высокую скорость Realtime Database для данных с частым обновлением (статусы дружбы, достижения) и гибкость запросов Firestore для аналитических и отображаемых данных.

Во-первых, анализ семи ведущих образовательных платформ подтвердил наличие незанятой ниши: отсутствует платформа, одновременно обеспечивающая промышленное качество автоматической проверки Java-кода, геймификацию через 3D/2D игровой движок Unity и открытую масштабируемую серверную архитектуру.
Во-вторых, исследование психологических основ геймификации показало, что системы достижений, уровней и социальных рейтингов напрямую удовлетворяют базовые психологические потребности, описанные в теории самодетерминации, что делает геймификацию научно обоснованным инструментом повышения мотивации к обучению.
В-третьих, сравнительный анализ технологий изоляции процессов (ptrace, SecurityManager, контейнеризация Docker) однозначно определил Docker как наиболее оптимальное решение для безопасного выполнения недоверенного Java-кода на сервере, обеспечивающее изоляцию на уровне ядра ОС при минимальных накладных расходах.
В-четвёртых, исследование технологий асинхронного взаимодействия определило Server-Sent Events (SSE) как наиболее подходящий протокол для реализации push-уведомлений в сторону Unity-клиента с учётом требований к надёжности, простоте масштабирования и совместимости с браузерными ограничениями WebGL-сборок.
Полученные выводы послужили основой для принятия ключевых архитектурных решений, описанных в последующих главах.
Проектирование архитектуры серверной части
Общая концепция распределённой архитектуры
Проектирование архитектуры программного комплекса является основополагающим этапом, на котором принимаются решения, определяющие долгосрочные характеристики системы: масштабируемость, отказоустойчивость, безопасность и стоимость сопровождения. Применительно к разрабатываемой образовательной платформе архитектурные решения диктуются двумя группами специфических требований.
К первой группе относятся требования, типичные для любого современного веб-сервиса: высокая доступность (желательно не менее 99.9% uptime), устойчивость к нагрузочным пикам (например, когда большая группа студентов одновременно сдаёт задание в конце занятия), безопасность пользовательских данных и логической целостности игрового прогресса.
Ко второй группе относятся требования, уникальные для данного класса систем: необходимость безопасного выполнения потенциально вредоносного кода в строгой изоляции от инфраструктуры сервиса, интеграция с клиентом на игровом движке Unity, функционирующим в виде WebGL-сборки в браузере, и необходимость работы push-уведомлений без постоянного опроса со стороны игрового клиента.
На основании этих требований была принята архитектура, построенная вокруг трёх физических уровней: клиентского (браузер пользователя), серверного (облачный Backend на Node.js) и облачной инфраструктуры (Firebase, Cloudinary). Граница между клиентским и серверным уровнями проходит строго по протоколу HTTPS: весь обмен данными осуществляется через зашифрованные запросы, что исключает перехват данных в транзитном сетевом трафике.
Ключевым архитектурным принципом является разделение ответственности (Separation of Concerns) на всех уровнях системы. На уровне кода это выражается в строгом разграничении слоёв маршрутизации, контроллеров и сервисов. На уровне инфраструктуры это проявляется в том, что подсистема выполнения Java-кода (Docker Sandbox) полностью изолирована от основного процесса Node.js: сбой или зависание Docker-контейнера не может нарушить работу остальных эндпоинтов API.
2.2 Детальное проектирование слоёв серверного приложения
2.2.1 Слой маршрутизации
Слой маршрутизации является входной точкой для всех HTTP-запросов, поступающих на сервер. В архитектуре Express.js этот слой реализован в виде набора Router-объектов, каждый из которых группирует эндпоинты одного функционального домена. Так, в проекте выделены следующие роутеры: auth.routes.js (регистрация и авторизация), user.routes.js (управление профилем), courses.routes.js (каталог курсов), lessons.routes.js (получение уроков), submissions.routes.js (отправка кода на проверку), gamification.routes.js (статистика и XP), achievements.routes.js (управление достижениями), leaderboard.routes.js (рейтинги), friends.routes.js (социальная система), notifications.routes.js (SSE-поток), news.routes.js (информационные блоки) и upload.routes.js (загрузка медиафайлов).
Каждый роутер подключается к центральному файлу index.js с назначением URL-префикса. Например, все запросы, начинающиеся с /api/v1/achievements, обрабатываются роутером достижений. Такая структура обеспечивает консистентность API и упрощает навигацию по кодовой базе.
На уровне маршрутизации также применяются middleware-компоненты. Middleware — это функции, которые выполняются в цепочке обработки запроса до передачи его контроллеру. В системе определены два ключевых middleware: verifyToken, проверяющий подлинность JWT-токена Firebase, и необязательный middleware проверки роли пользователя (adminOnly) для административных операций.
2.2.2 Слой контроллеров
Контроллеры образуют второй слой приложения. Их задача строго определена: принять объект HTTP-запроса (req), извлечь из него необходимые данные (параметры пути, query-параметры, тело запроса), выполнить базовую валидацию и передать управление соответствующему сервису. После получения результата от сервиса контроллер формирует структурированный JSON-ответ и устанавливает корректный HTTP-статус.
Важнейшим принципом является то, что контроллеры не содержат бизнес-логики. Они знают только о форматах HTTP и не знают о структуре базы данных или алгоритмах начисления очков. Это делает логику независимой от протокола: при необходимости добавить альтернативный способ взаимодействия (например, gRPC или GraphQL) потребуется создать только новый слой «входной точки», не трогая сервисы.
Все контроллеры используют паттерн обработки ошибок через блоки try-catch с последующей передачей ошибки в централизованный error handler Express. Централизованный обработчик ошибок стандартизирует формат всех ответов об ошибках: он включает числовой HTTP-статус, машиночитаемый код ошибки (error) и человекочитаемое сообщение (message). Это особенно важно для Unity-клиента, который должен уметь разбирать ошибочные ответы и отображать понятные пользователю сообщения в интерфейсе игры.
2.2.3 Слой контроллеров
Сервисный слой является ядром бизнес-логики приложения. Именно здесь сосредоточены все алгоритмы, связанные с семантикой предметной области: вычисление нового уровня пользователя, проверка условий выдачи достижений, формирование глобального рейтинга, управление жизненным циклом запросов в друзья.
В разрабатываемой системе ключевыми сервисами являются: achievements.service.js (полный жизненный цикл достижений), friends.service.js (управление социальными связями), leaderboard.service.js (расчёт рейтингов), auth.service.js (создание и валидация учётных данных), user.service.js (операции над профилями) и submissions.service.js (оркестрация процесса проверки кода).
Сервис достижений реализует сложную логику с тремя уровнями. На первом уровне хранится справочник всех доступных достижений в Firebase Realtime Database с описанием условий их получения (conditionType и conditionValue). На втором уровне функция checkAndAwardAchievements итерирует по всем достижениям и для каждого из них проверяет, выполнено ли соответствующее условие на основе актуальной статистики пользователя. Поддерживаются условия типа «решено N задач» (challenges_solved), «набрано N очков опыта» (total_xp), «достигнут N уровень» (level_reached), «серия N дней подряд» (streak_days) и ряд специфических игровых условий (например, прохождение уровня без подсказок или за установленное время). На третьем уровне функция awardAchievement атомарно записывает достижение в профиль пользователя, исключая дублирование через проверку существования записи перед записью.
Сервис друзей реализует конечный автомат состояний для отношений между пользователями. Каждое отношение может находиться в одном из трёх состояний: pending_sent (запрос отправлен текущим пользователем), pending_received (запрос получен от другого пользователя) или active (дружба установлена). При отправке запроса сервис одновременно записывает два симметричных узла в Firebase: для отправителя запись со статусом pending_sent, для получателя — с pending_received. При принятии запроса оба узла атомарно обновляются до статуса active. Удаление из друзей одновременно удаляет оба узла. Такая двунаправленная структура позволяет любому из участников получить полный список своих друзей и входящих запросов одним чтением своего раздела в базе данных, без необходимости выполнять дополнительные JOIN-операции.
2.3 Проектирование структуры данных и модели хранилища
2.3.1 Стратегия денормализации в NoSQL
В реляционных базах данных стандартной практикой является нормализация: данные разделяются на таблицы с минимальной избыточностью, а запросы к нескольким таблицам объединяются через операцию JOIN. В NoSQL-системах, в том числе в Firebase, операции JOIN отсутствуют, и потому стратегия проектирования данных строится иначе: данные моделируются исходя из паттернов чтения (read-driven design), а не из принципов нормализации.
Это означает, что часть данных намеренно дублируется в нескольких местах, чтобы обеспечить возможность получения нужного набора данных одним чтением. Например, в объекте записи о дружбе (friends/{userId}/{friendId}) помимо статуса отношения хранится временная метка (timestamp), позволяющая сортировать список друзей без дополнительного запроса. Профиль пользователя в Firestore содержит денормализованное поле stats с уже вычисленными агрегатными значениями (level, xp, completedLessons, achievementsCount), что исключает необходимость пересчитывать эти значения при каждом запросе профиля.
2.3.2 Проектирование схемы достижений
Достижения — одна из наиболее технически сложных подсистем с точки зрения проектирования данных. В отличие от простых свойств пользователя, достижения представляют собой двусторонние отношения: список всех существующих достижений в системе и подмножество из них, разблокированных конкретным пользователем.
Для хранения этих данных в Firebase Realtime Database используются две отдельные коллекции. Коллекция achievements содержит «каталог» всех достижений: для каждого из них хранится уникальный идентификатор, ссылка на категорию, локализованные строки названия и описания на двух языках (русском и английском), ссылки на иконки в двух состояниях (заблокировано и разблокировано), тип и целевое значение условия получения, а также размер XP-вознаграждения.
Коллекция userAchievements хранит данные о разблокированных достижениях каждого пользователя. Путь хранения организован как userAchievements/{userId}/{achievementId}. Помимо временной метки разблокирования, каждая запись содержит флаг notified, отслеживающий, было ли уведомление об этом достижении отправлено пользователю через SSE-поток. Также хранятся флаг isPinned и числовое поле pinOrder, реализующие функцию «витрины» — пользователь может выбрать до девяти наиболее ценных достижений для отображения на своей публичной странице профиля в игровом интерфейсе.
2.3.3 Проектирование схемы прогресса по островам и уровням
Образовательный контент в игровом модуле Unity организован иерархически: острова (Islands) содержат уровни-испытания (Challenges). Прогресс пользователя по этой иерархии хранится в специализированных коллекциях и запрашивается через специальные API-эндпоинты.
Для каждого острова в базе данных хранится объект прогресса, включающий: флаг разблокированности острова (unlocked), количество пройденных испытаний (completedChallenges), общее количество испытаний острова (totalChallenges) и количество набранных звёзд (stars). Количество звёзд рассчитывается на стороне сервера на основе качества прохождения: прохождение с первой попытки, без использования подсказок и за минимальное время даёт три звезды.
Для каждого отдельного испытания-уровня сервер хранит более детализированный объект прогресса: флаг прохождения (completed), количество попыток (totalAttempts и failedAttempts), количество использованных подсказок (hintsUsed), лучшее время прохождения (bestTime) и количество строк кода в лучшем решении (codeLines). Эти данные используются как для отображения в игровом интерфейсе Unity, так и для проверки сложных условий достижений типа «пройти уровень менее чем за 30 секунд» или «пройти 5 уровней подряд без сброса».
2.4 Детальное проектирование REST API
2.4.1 Принципы версионирования и структуры URL
Все маршруты API версионированы через префикс /api/v1/. Такой подход позволяет в будущем вводить новые версии API (v2, v3) с изменёнными контрактами, не нарушая работу уже развёрнутых клиентских приложений. Unity-клиент, встроенный в ранее опубликованные версии приложения, продолжает работать на /api/v1/, пока разработчики переводят более новые сборки на /api/v2/.
URL-адреса ресурсов строятся в соответствии с принципами REST: ресурсы именуются существительными во множественном числе (courses, achievements, friends), а действия над ними выражаются через HTTP-методы (GET — чтение, POST — создание, PUT/PATCH — обновление, DELETE — удаление). Для вложенных ресурсов применяется иерархическая структура URL (например, /courses/:courseId/lessons).
Полный формат ответа API стандартизирован: успешные ответы всегда содержат поля status (строка "success"), data (полезные данные) и, при необходимости, pagination (метаданные для постраничного вывода). Ошибочные ответы содержат status ("error"), code (машиночитаемый код) и message.
2.4.2 Система авторизации и middleware-цепочки
Безопасность API обеспечивается двухуровневой системой проверок. На первом уровне middleware verifyToken проверяет наличие заголовка Authorization: Bearer {token} в запросе, извлекает токен и вызывает метод admin.auth().verifyIdToken() из SDK firebase-admin. Этот метод криптографически верифицирует подпись токена с использованием публичных ключей Google Firebase, проверяет срок действия токена (exp claim) и соответствие аудитории (aud claim). Если проверка успешна, UID пользователя сохраняется в объекте запроса (req.uid) и передаётся дальше по цепочке. Если токен недействителен, возвращается ответ 401 Unauthorized с соответствующим сообщением.
На втором уровне отдельные административные эндпоинты проверяют поле role в профиле пользователя. Только пользователи с ролью admin могут создавать новые курсы, достижения и публиковать новости.
Такая архитектура делает систему авторизации полностью stateless: сервер не хранит никаких сессий или cookie — всё необходимое содержится в самом токене. Это упрощает горизонтальное масштабирование, поскольку любой экземпляр сервера может проверить любой токен, не обращаясь к центральному хранилищу сессий.
2.5 Проектирование подсистемы интеграции с Unity WebGL
2.5.1 Особенности среды выполнения Unity WebGL
Unity WebGL представляет собой принципиально иную среду по сравнению с нативными десктопными или мобильными сборками. Игра компилируется компилятором IL2CPP в WebAssembly, который выполняется внутри браузерного движка JavaScript. Это влечёт несколько фундаментальных ограничений.
Во-первых, WebGL-сборка не имеет доступа к файловой системе устройства. Все данные, которые игра хочет сохранить между сессиями, должны быть либо отправлены на сервер, либо записаны в браузерный localStorage. В разрабатываемой системе принято решение хранить весь прогресс исключительно на сервере, что обеспечивает доступность прогресса с любого устройства пользователя.
Во-вторых, сетевые запросы внутри WebGL-сборки выполняются не через стандартные системные сокеты, а через браузерный XMLHttpRequest API. Unity абстрагирует это через класс UnityWebRequest, однако поведение может отличаться от нативных платформ: в частности, некоторые HTTP-методы и заголовки могут быть ограничены браузерными политиками.
В-третьих, WebGL-сборка не может самостоятельно инициировать аутентификацию в Firebase: для этого требуется JavaScript API Firebase SDK, который недоступен внутри WebAssembly напрямую. Решением этой проблемы является описанный в следующем параграфе механизм передачи токена через postMessage.
2.5.2 Механизм авторизации Unity-клиента: проектное решение
Центральной архитектурной проблемой интеграции является необходимость передачи JWT-токена Firebase из веб-страницы-обёртки (написанной на React) в игру Unity, работающую внутри iframe. Прямой доступ к данным между фреймами заблокирован браузером политикой безопасности Same-Origin Policy.
Стандартным и безопасным способом межфреймового взаимодействия является API window.postMessage, специально введённый в стандарт HTML5 для этой цели. Механизм работы следующий: React-приложение получает актуальный токен от Firebase Authentication, используя метод getIdToken(), и вызывает метод postMessage на объекте contentWindow дочернего iframe. Сообщение имеет строго определённую структуру с полем type для идентификации вида сообщения и полем value для передаваемых данных.
На стороне Unity в папке Plugins размещается JavaScript-плагин (файл с расширением .jslib), который регистрирует обработчик события message на объекте window. При получении сообщения с типом token плагин вызывает метод SendMessage среды Unity, направляя полученную строку токена в указанный GameObject и метод C#-скрипта. Таким образом, строка токена попадает в поле firebaseIdToken объекта TokenManager и становится доступна всем сетевым методам игры.
Важным аспектом проектного решения является обработка ситуации истечения токена. Токены Firebase имеют срок действия один час. React-приложение периодически запрашивает у Firebase Authentication обновлённый токен (метод getIdToken с флагом forceRefresh) и повторно отправляет его в Unity через postMessage. Скрипт TokenManager в Unity при получении нового токена обновляет поле firebaseIdToken, при этом не запуская повторную загрузку данных сессии, если сессия уже была успешно инициализирована.
2.5.3 Синхронизация визуальных настроек
Помимо передачи токена, React-приложение синхронизирует ряд пользовательских настроек с игровым модулем Unity. В частности, передаются текущая тема оформления (тёмная или светлая) и язык интерфейса (русский или английский). Это позволяет адаптировать внешний вид интерфейса Unity-игры (цвет фона, цветовые схемы меню, язык игрового текста) без перезагрузки страницы, в реальном времени реагируя на изменения настроек пользователем.
2.6 Проектирование алгоритма начисления очков опыта и уровневой прогрессии
Одним из наиболее критически важных для игрового баланса аспектов является правильное проектирование кривой прогрессии — зависимости между накопленным опытом (XP) и текущим уровнем пользователя. Слишком пологая кривая (когда каждый следующий уровень требует лишь ненамного больше опыта, чем предыдущий) лишает систему чувства прогресса. Слишком крутая кривая (когда поздние уровни требуют многократно больше усилий) вызывает фрустрацию и отток пользователей.
В разрабатываемой системе принята квадратичная зависимость: порог XP для достижения уровня N вычисляется по формуле, в которой значение XP пропорционально квадрату номера уровня. Это обеспечивает сравнительно быстрое прохождение ранних уровней (создавая ощущение быстрого старта и мотивируя продолжение обучения) при постепенном нарастании требований для высоких уровней (создавая долгосрочную игровую цель).
Начисление XP происходит за несколько видов действий, каждому из которых назначена ценность, отражающая его образовательную значимость: прочтение теоретического урока, решение задачи типа quiz, успешное прохождение испытания-уровня в Unity, получение достижения (дополнительный бонус). Серверный сервис обновления прогресса выполняет атомарное увеличение поля xp с помощью Firebase ServerValue.increment, что исключает потерю данных при конкурентных запросах. После инкремента вычисляется новое значение уровня и, если оно изменилось по сравнению с текущим, поле level обновляется вместе с записью события для возможной выдачи достижения за достижение нового уровня.
2.7 Выводы по второй главе
В рамках второй главы была разработана полная архитектурная документация серверной части образовательного игрового приложения. Определена трёхуровневая структура приложения (маршруты, контроллеры, сервисы) с чётким разграничением ответственности каждого слоя. Спроектированы схемы хранения данных в Firebase Realtime Database и Firestore с применением стратегии денормализации, адаптированной под паттерны чтения конкретных функциональных модулей.
Разработан детализированный проект REST API с версионированием, стандартизированным форматом ответов и двухуровневой системой авторизации на основе JWT. Спроектирован механизм безопасной передачи учётных данных из React-приложения в Unity WebGL-сборку через postMessage API. Описаны алгоритмы ключевых компонентов бизнес-логики: системы достижений, социальной системы друзей и кривой прогрессии уровней.
3 Программная реализация и тестирование
3.1 Реализация подсистемы автоматической проверки Java-кода
3.1.1 Жизненный цикл проверки кода
Подсистема автоматической проверки является наиболее технически сложным и инженерно-значимым компонентом разработанного Backend-сервиса. Процесс проверки включает несколько строго последовательных этапов, каждый из которых реализован с учётом требований безопасности и надёжности.
На первом этапе (приём и валидация) сервер получает POST-запрос на эндпоинт /api/v1/submissions/run. Middleware verifyToken проверяет JWT и извлекает идентификатор пользователя. Контроллер выполняет валидацию тела запроса: проверяет наличие обязательных полей lessonId и code, а также то, что длина исходного кода не превышает установленного максимума (для предотвращения атак с передачей чрезмерно большого тела запроса). При несоответствии возвращается ответ 400 Bad Request.
На втором этапе (подготовка среды) сервисный модуль генерирует уникальный идентификатор задачи (UUID v4) и создаёт временную директорию по пути /tmp/{uuid}/. Из базы данных по идентификатору урока (lessonId) загружается объект испытания (challenge), содержащий тестовые случаи (testCases) и метаданные задания. Исходный код пользователя сохраняется в файл Main.java внутри временной директории. Тестирующий код формируется на основе testCases из базы данных и сохраняется рядом.
На третьем этапе (изолированное выполнение) Node.js-процесс выполняет системный вызов для запуска Docker-контейнера. Команда формируется динамически с набором параметров безопасности: флаг --rm для автоматического удаления контейнера после завершения, флаг --network none для отключения сетевого доступа, флаги --memory и --cpus для ограничения ресурсов через Cgroups, а также монтирование только временной директории задачи. Внутри контейнера последовательно выполняются компиляция (javac) и запуск тестов (java). Стандартный вывод и поток ошибок перехватываются для последующего анализа.
На четвёртом этапе (анализ результата и начисление опыта) сервер анализирует код завершения процесса и содержимое потоков вывода. Если код завершения равен нулю и стандартный вывод соответствует ожидаемому результату — задача считается решённой (статус Accepted). Ненулевой код завершения при наличии вывода компилятора javac в потоке ошибок интерпретируется как Compilation Error. Код завершения 137 означает принудительное завершение ядром Linux из-за превышения лимита памяти (Memory Limit Exceeded). Тайм-аут процесса на стороне Node.js интерпретируется как Time Limit Exceeded. При статусе Accepted сервис прогресса атомарно начисляет XP и обновляет статистику пользователя в Firebase.
На пятом этапе (очистка) временная директория /tmp/{uuid}/ рекурсивно удаляется вне зависимости от результата проверки. Это предотвращает накопление временных файлов на диске сервера. Docker-контейнер уже уничтожен автоматически благодаря флагу --rm.
3.1.2 Оптимизация производительности Judge-модуля
Наивная реализация, при которой для каждой проверки производится полный цикл Pull образа Docker → Create Container → Start Container, имеет неприемлемо высокую задержку (латентность). На практике образ openjdk:17-alpine (объёмом около 100 МБ) кэшируется локально после первого скачивания, что исключает повторную загрузку. Тем не менее, создание и запуск нового контейнера занимает от 200 до 500 миллисекунд.
Для снижения воспринимаемой задержки применяется несколько оптимизаций. Во-первых, выбор лёгкого базового образа Alpine Linux вместо полноценного дистрибутива сокращает размер слоёв контейнера и ускоряет его создание. Во-вторых, JVM при первом запуске выполняет инициализацию, занимающую значительное время. Применение флагов настройки JVM снижает накладные расходы инициализации в однопоточном режиме для простых задач.
В качестве перспективной оптимизации архитектура предусматривает возможность внедрения «горячего» пула контейнеров (warm container pool): набора заранее запущенных и ожидающих контейнеров с загруженной JVM, готовых немедленно принять код на выполнение. Такой подход применяется в промышленных Judge-системах и позволяет снизить латентность до нескольких десятков миллисекунд.
3.2 Реализация подсистемы Server-Sent Events
3.2.1 Архитектура SSE-сервиса на стороне Node.js
Server-Sent Events реализованы в форме выделенного сервиса (sse.service.js) и маршрута (notifications.routes.js). Сервис поддерживает реестр активных SSE-соединений: структуру данных типа Map, в которой ключом является UID пользователя, а значением — объект HTTP-ответа (res) Express.js с установленными заголовками потокового соединения.
При подключении Unity-клиента к эндпоинту GET /api/v1/notifications/stream сервер устанавливает специальные HTTP-заголовки (Content-Type: text/event-stream, Cache-Control: no-cache, Connection: keep-alive), регистрирует объект ответа в реестре соединений и периодически отправляет в поток специальные комментарии-пинги для предотвращения разрыва соединения таймаутами прокси-серверов и балансировщиков нагрузки.
Когда в системе происходит значимое событие (например, другой пользователь отправляет запрос в друзья), сервис получает UID целевого пользователя, находит его объект ответа в реестре соединений и записывает в поток сформатированное SSE-сообщение. Формат соответствует стандарту W3C: каждое событие начинается со строки data:, содержащей сериализованный JSON, и завершается двойным символом перевода строки.
При разрыве соединения (например, пользователь закрыл вкладку браузера) объект ответа автоматически удаляется из реестра соединений через обработчик события close, предотвращая утечки памяти при попытке записи в закрытый поток.
3.2.2 Реализация SSE-клиента в Unity (SSEClient.cs)
На стороне Unity реализация SSE-клиента представляет собой нетривиальную задачу. Стандартный класс UnityWebRequest предназначен для запросов с конечным ответом: он буферизует полученные данные и предоставляет их только после закрытия соединения сервером. Для бесконечного потока SSE такой подход неприменим.
Решением стало создание пользовательского наследника абстрактного класса DownloadHandlerScript, предоставляемого движком Unity для расширенной обработки загружаемых данных. В переопределённом методе ReceiveData, который вызывается ядром Unity синхронно при получении каждого нового фрагмента данных по сети, реализован инкрементальный построчный парсер SSE-потока. Парсер накапливает входящие байты в строковом буфере, ищет символы перевода строки (разделители SSE-сообщений) и извлекает законченные строки. Строки с префиксом data: обрабатываются как SSE-события: JSON-содержимое после префикса десериализуется, и на основании поля type вызывается соответствующий C#-делегат (Action), подписанный на нужный тип событий.
Корутина подключения к SSE реализована в виде цикла с автоматическим переподключением: при завершении текущего потокового запроса (по любой причине — сетевой разрыв, перезапуск сервера) выполняется пауза в несколько секунд и устанавливается новое соединение. Это обеспечивает устойчивость клиентского соединения к кратковременным сетевым сбоям.
3.3 Реализация сетевого модуля Unity (TokenManager)
3.3.1 Паттерн Singleton и управление жизненным циклом
TokenManager является центральным сетевым компонентом игрового приложения Unity. Он реализован как Singleton (паттерн проектирования, гарантирующий единственный экземпляр класса в приложении) с пометкой DontDestroyOnLoad, что позволяет объекту сохраняться при переходах между игровыми сценами (например, с главного экрана на экран выбора острова или в сцену конкретного уровня).
При инициализации TokenManager выполняет поиск и кэширование ссылок на все UI-панели, с которыми ему предстоит взаимодействовать: панель статистики (StatsPanelBinder), панель достижений (AchievementsPanelBinder), таблица лидеров (LeaderboardPanelBinder), список друзей (FriendsPanelBinder) и панель закреплённых достижений (PinnedAchievementsPanelBinder). Кэширование ссылок устраняет необходимость выполнять дорогостоящие операции поиска объектов (FindObjectByType) при каждом обновлении данных.
3.3.2 Паттерн Singleton и управление жизненным циклом
Процесс загрузки данных при начале игровой сессии реализован в виде корутины LoadAllSessionData. Корутины в Unity — это методы-сопрограммы, которые могут приостанавливать своё выполнение в ожидании завершения асинхронных операций (таких как HTTP-запросы), не блокируя при этом основной поток рендеринга. Это принципиально важно: заморозка основного потока привела бы к «зависанию» картинки на экране, что неприемлемо с точки зрения пользовательского опыта.
Корутина LoadAllSessionData последовательно выполняет десять запросов к Backend API: получение профиля пользователя, статистики геймификации, справочника категорий достижений, полного каталога достижений, достижений пользователя, таблицы лидеров, списка друзей, прогресса по островам и прогресса по отдельным испытаниям-уровням. Запросы выполняются последовательно (каждый следующий ждёт завершения предыдущего), поскольку данные из ранних запросов могут потребоваться для формирования запросов последующих (например, UID пользователя из профиля используется в запросе статистики).
После завершения всех запросов TokenManager вызывает методы Apply на соответствующих UI-binder компонентах, передавая им загруженные данные. Binder-компоненты отвечают за визуальное отображение данных в UI-элементах Unity (TextMeshPro, Image, ListView и т.д.). Флаг IsSessionReady устанавливается в true, что разблокирует интерактивные элементы игрового меню.
3.3.3 Обобщённый метод выполнения GET-запросов
Для устранения дублирования кода при выполнении множества типизированных GET-запросов разработан обобщённый метод-корутина с параметром типа T. Метод принимает относительный путь к ресурсу API и коллбэк-делегат, вызываемый при успешном получении ответа. Внутри метода формируется полный URL путём конкатенации базового URL API (apiBaseUrl) с относительным путём, создаётся UnityWebRequest с установленным заголовком авторизации, выполняется запрос и после получения ответа выполняется попытка десериализации JSON через JsonUtility.FromJson с передачей результата в коллбэк. Обработка ошибок (сетевые сбои, ошибки HTTP, ошибки парсинга JSON) выполняется внутри метода с выводом диагностических сообщений в консоль Unity.
Аналогичные обобщённые вспомогательные методы реализованы для других HTTP-методов: PostEmpty (POST без тела), PostJson (POST с JSON-телом), PutJson (PUT с JSON-телом) и DeleteRequest. Каждый из них принимает путь и коллбэк типа Action с двумя параметрами: булевым флагом успеха и строкой сообщения об ошибке.
3.4 Тестирование программного комплекса
3.4.1 Тестирование безопасности подсистемы изоляции кода
Проверка корректности работы Docker Sandbox является наиболее критичным аспектом тестирования всего программного комплекса. Для этого разработана серия тест-кейсов, имитирующих действия злонамеренного пользователя, стремящегося вывести сервер из строя или получить несанкционированный доступ к данным.
Тест 1: Бесконечный цикл. Пользовательский код содержит бесконечный цикл без условия выхода. Ожидаемый результат: по истечении установленного тайм-аута (10 секунд) Node.js завершает дочерний процесс Docker принудительно, сервер возвращает клиенту статус Time Limit Exceeded. Фактический результат: тест пройден, нагрузка на процессор хост-машины после завершения тест-кейса немедленно вернулась к норме, что подтверждает корректное уничтожение контейнера.
Тест 2: Исчерпание памяти. Пользовательский код аллоцирует массивы до исчерпания доступной памяти. Ожидаемый результат: при достижении лимита в 256 МБ ядро Linux принудительно завершает процесс JVM (OOM Killer), контейнер завершается с кодом 137, сервер возвращает Memory Limit Exceeded. Фактический результат: тест пройден, объём свободной оперативной памяти хост-машины не изменился.
Тест 3: Обращение к файловой системе. Код пытается прочитать файл /etc/passwd хост-машины. Ожидаемый результат: внутри контейнера существует изолированная файловая система Alpine Linux; код успешно читает файл /etc/passwd контейнера (содержащий стандартные системные записи Alpine), но не получает никаких данных хост-системы, включая переменные окружения с ключами Firebase. Фактический результат: тест пройден. Данные хост-машины остались недоступны.
Тест 4: Сетевые соединения. Код пытается открыть TCP-соединение с внешним сервером. Ожидаемый результат: флаг --network none полностью отключает сетевой стек контейнера; попытка создания сокета вызывает исключение SocketException с сообщением «Network is unreachable». Фактический результат: тест пройден.
3.4.2 Нагрузочное тестирование REST API
Для оценки производительности API при одновременной работе многочисленных пользователей была применена утилита Apache JMeter. Тестовый план включал несколько сценариев с различной нагрузкой.
Сценарий A (GET-запросы профилей): 200 виртуальных пользователей одновременно запрашивают свои профили. Среднее время ответа составило 85 мс, 95-й перцентиль — 210 мс, ошибок не зафиксировано. Node.js успешно справился с нагрузкой благодаря асинхронной модели Event Loop.
Сценарий B (SSE-соединения): 500 виртуальных пользователей устанавливают соединение с SSE-потоком и удерживают его 2 минуты. Потребление оперативной памяти процессом Node.js увеличилось пропорционально числу соединений (примерно 160 КБ на каждое соединение). Утечек памяти при завершении соединений не обнаружено: после отключения всех клиентов потребление памяти вернулось к исходному значению.
Сценарий C (проверка кода): 50 одновременных запросов на проверку Java-кода. Поскольку каждый запрос порождает отдельный Docker-контейнер, нагрузка на ЦПУ сервера резко возрастала. При недостатке вычислительных ресурсов запросы ставились в очередь на уровне Node.js Event Loop. Время обработки запроса при 50 одновременных пользователях составило от 3 до 12 секунд, что находится в допустимых пределах для образовательного приложения. Все 50 запросов были корректно обработаны без ошибок сервера.
3.4.3 Тестирование корректности логики геймификации
Отдельную категорию составляют тесты бизнес-логики начисления опыта и выдачи достижений. Для этих тестов применён подход чёрного ящика: тестируются входные и выходные данные без анализа внутренней реализации.
Тест на идемпотентность: одно и то же завершённое испытание отправляется на сервер дважды. Ожидаемый результат: XP начисляется только один раз. Фактический результат: тест пройден. Сервис прогресса перед начислением XP проверяет, не было ли данное испытание уже завершено ранее.
Тест на корректность выдачи достижений: пользователю искусственно устанавливается значение статистики, соответствующее условию получения достижения (например, значение completedChallenges = 10). После вызова функции проверки достижений выполняется запрос к базе данных. Ожидаемый результат: в разделе пользователя в userAchievements появилась запись о полученном достижении. Фактический результат: тест пройден по всем проверяемым типам условий (challenges_solved, total_xp, level_reached, streak_days).
Тест на закрепление достижений: пользователь закрепляет более 9 достижений. Ожидаемый результат: при попытке закрепить десятое достижение сервер возвращает ошибку 400 Bad Request с сообщением «Max 9 pinned achievements allowed». Фактический результат: тест пройден.
3.5 Тестирование программного комплекса
В третьей главе детально описана практическая реализация всех ключевых компонентов разработанного программного комплекса. Описан пятиэтапный жизненный цикл проверки Java-кода в Docker-контейнере с механизмами защиты от всех ключевых векторов атак. Описана реализация SSE-сервиса на стороне Node.js и нестандартного SSE-клиента на стороне Unity с использованием пользовательского DownloadHandlerScript. Описана архитектура сетевого модуля TokenManager, реализующего паттерн Singleton, управление состоянием сессии через корутины и обобщённые HTTP-методы.
Результаты тестирования подтвердили: подсистема изоляции кода надёжно защищает сервер от всех четырёх протестированных векторов атаки; API выдерживает нагрузку в 200 одновременных пользователей без деградации качества обслуживания; SSE-инфраструктура не имеет утечек памяти при динамическом подключении и отключении клиентов; бизнес-логика геймификации корректно обрабатывает граничные случаи.
4 Реализация социальной системы, рейтингов и системы достижений
4.1 Концептуальная модель социальной платформы в образовательном контексте
Ключевым психологическим механизмом, удерживающим пользователей в образовательных приложениях, является социальная вовлечённость. Обучение в изоляции, без возможности сравнить свои успехи с другими и получить признание от сверстников, значительно менее эффективно, чем обучение в социальном контексте. Именно поэтому разработанная система включает развитый социальный модуль, который формирует вокруг образовательной платформы устойчивое сообщество.
Социальный модуль разрабатываемой системы охватывает три взаимосвязанных аспекта. Первый аспект — система межличностных связей (дружба), позволяющая пользователям находить и добавлять друг друга, формируя персонализированную социальную сеть внутри платформы. Второй аспект — многомерная система рейтингов (Leaderboard), предоставляющая различные срезы соревновательной статистики в зависимости от интересов и целей пользователя. Третий аспект — система достижений с публичным профилем, позволяющая пользователям демонстрировать свои успехи.
Принципиально важным проектным решением является то, что все три аспекта социального модуля полностью реализованы на стороне Backend-сервера и никогда не доверяют данным, присланным клиентом. Клиент (Unity или React) может лишь инициировать действие (отправить запрос в друзья, попросить показать таблицу лидеров), но не может самостоятельно изменить свой рейтинг или получить достижение.
4.2 Система управления социальными связями (Friendship System)
4.2.1 Идентификация пользователей: система дискриминаторов
Одной из практических проблем любой социальной системы является поиск конкретного пользователя среди потенциально большого числа участников с одинаковыми именами. В разрабатываемой системе эта проблема решена путём введения дискриминатора — четырёхзначного числового суффикса, автоматически назначаемого при регистрации.
Таким образом, уникальный «адрес» пользователя в системе имеет вид НикнеймПользователя#XXXX. Комбинация имени и дискриминатора гарантированно уникальна в рамках системы, что позволяет однозначно идентифицировать пользователя при отправке запроса в друзья.
На уровне базы данных поиск по паре (displayName, discriminator) реализован в сервисе getUserByUsernameAndDiscriminator, который сканирует коллекцию пользователей и возвращает первое совпадение. Поскольку оба поля хранятся в профиле пользователя, этот запрос не требует дополнительных структур индексирования.
4.2.2 Конечный автомат состояний дружбы
Жизненный цикл отношений между двумя пользователями описывается конечным автоматом с тремя состояниями. Изначально никакой связи между пользователями нет (состояние «отсутствует»). После того как один из пользователей отправляет запрос, система переходит в промежуточное состояние ожидания (для отправителя это pending_sent, для получателя — pending_received). Принятие запроса переводит отношения в активное состояние (active). Удаление из друзей или отклонение запроса возвращает систему в начальное состояние.
Особого внимания заслуживает обработка ситуации, когда пользователь B отправляет запрос пользователю A в то время, когда у пользователя A уже есть исходящий запрос к пользователю B (то есть пользователи одновременно нашли друг друга). В текущей реализации такая ситуация обрабатывается как ошибка с сообщением «Friend request already received from this user», что сигнализирует пользователю A о необходимости принять уже имеющийся входящий запрос, а не отправлять новый.
4.2.3 Атомарность операций в Firebase при управлении дружбой
Каждая операция изменения статуса дружбы затрагивает данные двух пользователей одновременно. При отправке запроса необходимо одновременно создать запись для отправителя и для получателя. При принятии запроса необходимо одновременно обновить обе записи до статуса active. Если эти операции выполнять последовательно и между ними произойдёт сбой сети или сервера, база данных окажется в несогласованном состоянии (например, у отправителя статус pending_sent, а у получателя запись отсутствует).
Для решения этой проблемы применяется механизм мультипутевых обновлений Firebase Realtime Database (Multi-Path Updates). Вместо двух последовательных вызовов update выполняется один вызов, в котором объект обновлений содержит записи для обоих пользователей. Firebase гарантирует атомарное применение всего объекта: операция либо успешно записывает все изменения, либо не записывает ни одного. Это обеспечивает консистентность данных при любых сбоях.
4.2.4 Push-уведомления через SSE при получении запроса в друзья
Немедленное информирование пользователя о входящем запросе в друзья является важным элементом пользовательского опыта. Именно для этого сценария в системе используется механизм Server-Sent Events.
После успешной записи обоих узлов в Firebase сервис дружбы вызывает функцию sendToUser сервиса SSE, передавая ей UID получателя и JSON-объект с описанием события (тип FRIEND_REQUEST и идентификатор отправителя). Сервис SSE находит открытое HTTP-соединение получателя в реестре соединений и записывает в поток событий сформированное сообщение. Unity-клиент получателя, постоянно слушающий этот поток через SSEClient.cs, немедленно реагирует на поступившее событие — отображает уведомление в UI и обновляет счётчик входящих запросов.
4.3 Многомерная система таблиц лидеров
4.3.1 Архитектура многомерного рейтинга
Разработанная система рейтингов принципиально отличается от большинства аналогов тем, что предоставляет не одну, а шесть различных таблиц лидеров, каждая из которых измеряет разный аспект достижений пользователя. Это расширяет аудиторию соревновательного контента: пользователь, не занимающий высоких позиций по накопленному опыту, может оказаться лидером по длине непрерывной серии (streak) или по количеству решённых задач одной из категорий сложности.
Реализованные таблицы лидеров и их критерии сортировки:
Глобальная по XP (getGlobalLeaderboard): первичная сортировка по накопленному опыту (xp), вторичная — по уровню (level), третичная — по количеству решённых задач (totalChallengesSolved). Многоуровневая сортировка исключает ситуацию «ничьей», когда два пользователя имеют одинаковый XP
По решённым задачам (getChallengesLeaderboard): первичная сортировка по числу пройденных испытаний-уровней. Отражает активность пользователя в игровом модуле Unity
По уровню (getLevelLeaderboard): отражает «глубину» прогресса пользователя в игровом контенте
По серии (getStreakLeaderboard): первичная сортировка по currentStreak (текущей непрерывной серии дней активности), вторичная
по longestStreak (рекорду серии). Стимулирует ежедневное использование платформы.
По курсу (getCourseLeaderboard): рейтинг пользователей в рамках конкретного учебного курса, основанный на проценте прохождения курса. Позволяет студентам одной учебной группы соревноваться в рамках общей программы.
По сложности (getDifficultyLeaderboard): рейтинг по числу решённых задач определённой категории сложности (beginner, intermediate, advanced). Даёт специализированный срез для пользователей, сосредоточенных на конкретной ступени сложности.
4.3.2 Алгоритм расчёта рейтинга и оптимизация
Базовый алгоритм получения любой из таблиц лидеров следует единой схеме: получить всех пользователей со статистикой, применить специфическую функцию сортировки, выполнить постраничную выборку (pagination) по параметрам limit и offset, добавить к каждому элементу его позицию в рейтинге (rank), рассчитываемую как offset + index + 1.
Наиболее вычислительно затратным этапом является получение всех пользователей через однократное чтение всей коллекции users из Firebase. При большом числе зарегистрированных пользователей (тысячи и более) эта операция может занимать значительное время и нести высокую стоимость по тарификации Firebase (каждый документ считывается отдельно). Для сглаживания этой нагрузки сервис leaderboard кэширует результаты глобального рейтинга в оперативной памяти процесса Node.js с заданным TTL (Time To Live). Повторные запросы в течение периода TTL немедленно получают ответ из кэша без обращения к Firebase, что многократно снижает как задержку ответа, так и расходы на операции чтения.
Для таблицы лидеров по курсам (getCourseLeaderboard) и по сложности (getDifficultyLeaderboard) применяется техника параллельного выполнения запросов (Promise.all). Вместо последовательного получения прогресса для каждого пользователя все запросы отправляются одновременно. Node.js, работая в рамках своей асинхронной модели, управляет параллельными запросами в пределах возможностей Event Loop, что значительно сокращает общее время ожидания.
4.3.3 Определение позиции конкретного пользователя
Важной дополнительной функцией является возможность узнать точное положение конкретного пользователя в рейтинге без необходимости скачивать весь список (метод getUserRank). Этот метод помимо позиции (rank) и общего числа участников (totalUsers) возвращает информацию о «соседях по рейтингу»: профиль и показатели пользователя, стоящего на позицию выше (above) и на позицию ниже (below). Отображение «соседей» является стандартной игровой практикой, создающей ощущение конкретного, достижимого соперника и усиливающей мотивацию к улучшению позиции.
4.4 Многоуровневая система достижений
4.4.1 Классификация достижений по типам условий
Разработанная система поддерживает богатое множество типов условий, автоматически проверяемых на сервере. Условия разделяются на несколько семантических групп.
Группа образовательного прогресса охватывает условия, связанные с количеством выполненных учебных действий: challenges_solved (решено N задач), lessons_completed (завершено N уроков), total_xp (накоплено N очков опыта), level_reached (достигнут N уровень). Эта группа стимулирует планомерное продвижение по учебной программе.
Группа постоянства охватывает условия, связанные с регулярностью занятий: streak_days (серия N дней подряд). Ежедневная активность является ключевым предиктором успешного завершения образовательного курса, поэтому её стимулирование через систему достижений имеет прямой педагогический эффект.
Группа мастерства охватывает специфические игровые достижения, выходящие за рамки простого подсчёта: прохождение уровня за минимальное время (level_time_under_30s, level_time_under_15s), прохождение уровней без использования подсказок (levels_without_hints), прохождение уровней без единого сброса позиции (levels_without_reset), решение задачи минимальным объёмом кода (code_lines_under_5), прохождение уровня после многочисленных провальных попыток (level_after_5_fails, level_after_10_fails). Эта группа достижений наиболее трудоёмка в разработке, но создаёт долгосрочные цели для опытных пользователей.
Группа активности охватывает достижения за нестандартное поведение: прохождение уровня в ночное время (level_after_23), прохождение до утра (level_before_7), повторное прохождение уже завершённого уровня (replay_completed_level) и суммарное время, проведённое в игре (total_playtime_hours).
4.4.2 Система закрепления достижений
Возможность закрепления достижений на публичном профиле реализует социальный аспект самовыражения, хорошо знакомый пользователям игровых платформ (Steam, Xbox, PlayStation). Каждый пользователь может выбрать до девяти наиболее ценных для него достижений и разместить их на «витрине» своего профиля, видимой другим пользователям при просмотре публичного профиля в игре.
Технически закрепление реализовано через два атрибута в записи о разблокированном достижении: булевый флаг isPinned и числовое поле pinOrder, задающее порядок отображения закреплённых достижений. При закреплении нового достижения сервер вычисляет следующее значение pinOrder как максимальное существующее значение pinOrder среди всех закреплённых плюс единица. Это обеспечивает добавление нового достижения в конец «витрины».
Ограничение в девять закреплённых достижений введено по нескольким причинам. С технической точки зрения, это ограничивает объём данных, передаваемых Unity-клиенту при загрузке публичного профиля. С дизайнерской точки зрения, ограниченное пространство «витрины» создаёт игровую задачу выбора: пользователь должен решить, какие из своих достижений наиболее важны, что само по себе является формой рефлексии над своим прогрессом.
Функция перестановки закреплённых достижений (reorderPinnedAchievements) позволяет пользователю изменить порядок отображения уже закреплённых достижений, отправив массив их идентификаторов в новом желаемом порядке. Сервер применяет новый порядок через мультипутевое обновление Firebase.
4.4.3 Локализация системы достижений
Разработанная система изначально ориентирована на двуязычную аудиторию: приложение поддерживает интерфейс на русском и английском языках, а Unity-клиент получает актуальный параметр языка от React-приложения через механизм postMessage. Для обеспечения корректного отображения достижений на нужном языке каждое достижение хранит поля title и description в виде объектов с ключами en и ru. При отображении в Unity-клиенте компонент PinnedAchievementsPanelBinder выбирает нужный языковой вариант на основе текущей локали приложения.
4.5 Система управления мультимедийным контентом
Профили пользователей поддерживают пользовательские аватары. Загрузка изображений является нетривиальной задачей для Backend-сервиса: необходимо принять бинарный файл от клиента, проверить его тип и размер, оптимизировать для хранения и передачи, разместить в надёжном облачном хранилище и сохранить URL в профиле пользователя.
Для хранения изображений используется платформа Cloudinary, специализирующаяся на управлении медиафайлами. Cloudinary выполняет автоматическую оптимизацию изображений «на лету»: при запросе аватара с параметрами размера (например, 100x100 пикселей) Cloudinary самостоятельно масштабирует изображение на своих серверах и кэширует его в CDN-сети (Content Delivery Network), распределённой по всему миру. Это означает, что аватар пользователя загружается к нему из ближайшего географически узла, минимизируя задержку.
На стороне Backend маршрут upload.routes.js принимает multipart/form-data запрос с помощью middleware multer, который обрабатывает загрузку файла в память Node.js. После успешной загрузки файл передаётся в API Cloudinary, которая возвращает постоянный URL загруженного изображения. URL сохраняется в поле photoURL профиля пользователя в Firebase.
4.6 Выводы по четвёртой главе
В четвёртой главе подробно описаны социальные и игровые подсистемы разработанного Backend-сервиса. Рассмотрен конечный автомат состояний системы дружбы с атомарными мультипутевыми обновлениями Firebase. Описана архитектура шести различных таблиц лидеров с оптимизацией через кэш оперативной памяти и параллельные запросы. Детально охарактеризована многоуровневая система достижений с двадцатью типами условий и функцией публичной «витрины» закреплённых достижений с поддержкой двух языков.
5 Система геймификации: прогресс, балансировка и развёртывание
5.1 Архитектура сервиса геймификации
Сервис геймификации (gamification.service.js) является самым объёмным и сложным по логике модулем всего Backend-сервиса — он содержит свыше 690 строк кода и оркестрирует взаимодействие большинства других сервисов: пользовательского (user.service.js), достижений (achievements.service.js) и прогресса (user-progress.service.js). Фактически этот сервис является ядром бизнес-логики образовательной платформы и координирует все игровые события.
Архитектурно сервис разделён на несколько функциональных блоков. Первый блок — инициализация и поддержание инвариантов: функция ensureUserStats гарантирует, что при первом обращении к статистике любого пользователя она будет корректно инициализирована нулевыми значениями. Это предотвращает ошибки NullPointerException при обращении к полям статистики нового пользователя. Второй блок — вычисление производных показателей: функции calculateLevel и calculateXP реализуют математику игрового баланса. Третий блок — обработка событий: функции awardXP, awardBadge, updateStreak, markLessonCompleted, markChallengeSolved реагируют на конкретные игровые события. Четвёртый блок — запросы аналитики: функции getUserProgress, getCourseProgress, getDetailedProgress, getLessonProgress предоставляют агрегированные данные прогресса по различным срезам.
5.2 Математика игрового баланса: формула XP и кривая уровней
5.2.1 Многоуровневое начисление очков опыта
Система начисления XP в разработанном приложении является многоуровневой — итоговая награда зависит от нескольких факторов, что создаёт различные стратегии поведения пользователей.
Базовая награда за прохождение испытания-уровня рассчитывается как произведение количества полученных звёзд на фиксированную ставку (50 XP за звезду). Таким образом, минимальная награда за прохождение — 50 XP (одна звезда), максимальная базовая награда — 150 XP (три звезды).
Бонус за скорость прохождения (Speed Bonus) начисляется дополнительно к базовой награде. Прохождение испытания быстрее 30 секунд даёт максимальный бонус скорости (+50 XP), прохождение за 30–60 секунд — половинный бонус (+30 XP). Это стимулирует оптимизацию алгоритма решения.
Бонус за самостоятельность (Hint-Free Bonus) начисляется, если пользователь прошёл испытание без использования ни одной подсказки (+30 XP). Этот бонус вступает в конфликт с другим типом поведения — пользователи, стремящиеся максимизировать скорость, часто предпочитают воспользоваться подсказкой, однако теряют бонус самостоятельности.
Таким образом, максимальная теоретическая награда за одно идеально пройденное испытание составляет 230 XP (150 + 50 + 30), что создаёт значительный стимул к мастерству.
5.2.2 Мультипликаторы сложности
Помимо базовой формулы, в системе используется коэффициент сложности (difficulty multiplier). Все испытания разделены на три категории сложности: beginner (начинающий), intermediate (средний) и advanced (продвинутый). Каждой категории назначен множитель из массива DIFFICULTY_MULTIPLIERS, определённого в файле констант gamification constants.
Применение множителя делает более сложные задачи не просто «необязательными» с точки зрения прогресса, а действительно выгодными: пользователь, решающий задачи категории advanced, накапливает XP значительно быстрее, что ускоряет повышение его уровня. Это создаёт ценностное предложение для опытных пользователей, которые стремятся к высоким уровням, а не просто к завершению образовательной программы.
5.2.3 Нелинейная кривая уровней
Кривая зависимости уровня от накопленного XP определяется таблицей порогов, хранящейся в объекте LEVELS. Первые десять уровней определены пороговыми значениями явно, что позволяет гейм-дизайнеру точно настраивать ощущение «темпа» прогрессии: ранние уровни проходятся быстро (создавая эффект быстрого старта), а каждый следующий уровень требует всё больше XP.
Для уровней выше десятого применяется линейная формула: каждые дополнительные 500 XP сверх порога десятого уровня дают ещё один уровень. Это создаёт «бесконечную» систему уровней, не ограничивающую прогресс самых активных пользователей.
5.3 Система ежедневной активности (Daily Streak)
5.3.1 Алгоритм обновления серии
Система «ежедневной серии» (streak) является одним из наиболее психологически эффективных механизмов удержания пользователей в любом приложении, требующем регулярного использования. Принцип прост: если пользователь проявлял активность в приложении вчера и сегодня, его серия увеличивается на единицу; если пользователь пропустил день — серия сбрасывается до единицы.
На уровне реализации алгоритм сравнивает текущую дату (нормализованную до начала дня, то есть 00:00:00) с датой последней активности пользователя. Нормализация до начала дня критически важна: без неё пользователь, игравший в 23:55 и в 00:05 следующего дня (то есть с разницей всего в 10 минут), не получил бы увеличения серии, так как разница в миллисекундах составила бы менее 24 часов.
Алгоритм различает три ситуации: первая активность когда-либо (устанавливает серию в 1), активность в тот же день (серия не изменяется, XP за серию не начисляется — защита от многократного фарма бонуса за серию в течение одного дня), активность после перерыва ровно в один день (серия увеличивается, начисляется бонус XP за поддержание серии), активность после перерыва более одного дня (серия сбрасывается до 1).
5.3.2 Сохранение рекорда серии
Помимо текущей серии (currentStreak), система сохраняет исторический максимум (longestStreak). Рекорд используется в таблице лидеров по серии (getStreakLeaderboard) как вторичный критерий сортировки: при одинаковых текущих сериях побеждает тот, чей исторический максимум выше. Это делает рекорд серии постоянной «реликвией» достижений, значимой даже после того, как текущая серия была прервана.
5.4 Система звёзд и оценки качества прохождения
Каждое испытание-уровень в Unity оценивается по трёхзвёздочной системе, широко применяемой в мобильных играх (Angry Birds, Clash of Clans и др.). Количество заработанных звёзд рассчитывается на стороне сервера на основе нескольких параметров, переданных Unity-клиентом по завершении уровня: времени прохождения (completionTime в секундах), количества неудачных попыток до успеха (failedAttempts), количества использованных подсказок (hintsUsed) и количества строк кода в итоговом решении (codeLines).
Наличие поля codeLines особенно интересно с педагогической точки зрения: поощрение за решение задачи минимальным объёмом кода стимулирует студентов думать об элегантности и лаконичности алгоритма, а не просто добиваться правильного результата любыми средствами.
Все эти данные сохраняются в детализированном объекте прогресса конкретного испытания в Firebase, что позволяет отображать их в профиле игрока в Unity и использовать для проверки условий специфических достижений.
5.5 Детализированная аналитика прогресса
5.5.1 Многоуровневая аналитика getCourseProgress
Функция getCourseProgress предоставляет исчерпывающую аналитику по конкретному учебному курсу для конкретного пользователя. Аналитика включает: общее число уроков курса и число завершённых; общее число испытаний курса и число решённых; процент завершения курса по урокам и по испытаниям (два отдельных показателя, так как урок можно завершить теоретически, не решив игровое испытание); ссылку на следующий не пройденный урок (nextLesson) для функции «Продолжить с того места»; флаг полного завершения курса.
5.5.2 Детальная статистика getDetailedProgress
Функция getDetailedProgress формирует полный «отчёт успеваемости» пользователя по всем курсам и всем уровням сложности. Этот же метод используется в сервисе leaderboard при построении рейтинга по сложности (getDifficultyLeaderboard).
Отчёт организован по двум измерениям. Первое измерение — по типу активности (уроки vs испытания). Для каждого типа сообщается общее количество, количество завершённых/решённых и разбивка по трём категориям сложности (beginner, intermediate, advanced). Второе измерение — по курсам: для каждого курса возвращается полный объект прогресса (как из getCourseProgress). Завершает отчёт сводный показатель общего прогресса (overallProgress) — процент завершённых уроков и процент решённых испытаний относительно суммарного контента платформы.
5.6 Развёртывание и конфигурирование серверной инфраструктуры
5.6.1 Конфигурирование переменных окружения
Backend-сервис содержит ряд конфиденциальных данных, которые категорически недопустимо хранить в системе контроля версий (Git): приватные ключи Firebase Admin SDK, API-ключ Cloudinary, URL базового сервиса. Все эти данные выносятся в переменные окружения (environment variables), управляемые через файл .env.
Файл .env добавлен в список исключений .gitignore, что гарантирует его отсутствие в репозитории. На продуктивном сервере переменные окружения задаются непосредственно в панели управления хостингового провайдера (в данном случае Render.com), что является стандартной индустриальной практикой.
При запуске сервис читает переменные окружения с помощью библиотеки dotenv и использует их для инициализации SDK: admin.initializeApp с serviceAccount из переменных, конфигурации Cloudinary и параметров URL.
5.6.2 Платформа развёртывания
Для развёртывания Backend-сервиса выбрана платформа Render.com — современный облачный провайдер категории PaaS (Platform as a Service). В отличие от классических VPS-хостингов (Virtual Private Server), где разработчик самостоятельно управляет операционной системой, Nginx и процесс-менеджером, PaaS-платформы автоматизируют инфраструктурный уровень.
Render.com автоматически распознаёт Node.js-проект по наличию файла package.json, устанавливает зависимости командой npm install, запускает сервер командой npm start (которая выполняет node src/server.js) и предоставляет зашифрованный HTTPS-эндпоинт с автоматическим обновлением SSL-сертификата. При каждом новом коммите в ветку main репозитория GitHub Render.com автоматически выполняет переразвёртывание (re-deployment), что реализует базовый CI/CD (Continuous Integration / Continuous Deployment) пайплайн без дополнительной настройки.
5.6.3 Конфигурация CORS для взаимодействия с Unity и React
Один из нетривиальных аспектов развёртывания — корректная настройка политики CORS (Cross-Origin Resource Sharing). Браузер блокирует AJAX-запросы к домену, отличному от того, с которого загружена страница, если сервер не явно разрешил такие запросы.
В разрабатываемой системе React-приложение и Unity WebGL-модуль могут быть развёрнуты на отличных от API-сервера доменах. Для корректной работы в конфигурации CORS указывается список разрешённых источников (origins), включающий домен React-приложения (firebase hosting), локальный адрес разработки (localhost с разными портами) и домен Unity WebGL-хостинга (если они различаются).
Дополнительно настраиваются разрешённые HTTP-заголовки (в том числе Authorization для передачи токена), разрешённые методы (GET, POST, PUT, DELETE, OPTIONS) и обработка предполётных запросов (Preflight, метод OPTIONS), которые браузер автоматически отправляет перед каждым «небезопасным» кросс-доменным запросом.
5.7 Выводы по пятой главе
В пятой главе подробно описана архитектура центрального сервиса геймификации, включая многоуровневую формулу начисления XP с мультипликаторами сложности, нелинейную кривую уровней и алгоритм обновления ежедневной серии. Рассмотрена система трёхзвёздочной оценки качества прохождения испытаний. Описаны функции многоуровневой аналитики прогресса, формирующие основу для построения специализированных таблиц лидеров. Освещены практические аспекты развёртывания: управление конфиденциальными данными через переменные окружения, автоматическое CI/CD развёртывание на платформе Render.com и настройка CORS.
ЗАКЛЮЧЕНИЕ
В ходе выполнения выпускной квалификационной работы была решена актуальная и технически сложная инженерная задача: спроектирована и реализована надежная серверная инфраструктура (Backend) для игрового образовательного приложения, а также разработан комплексный модуль сетевого взаимодействия на стороне игрового движка Unity.
В процессе реализации были достигнуты следующие результаты.
Анализ предметной области: Изучены существующие алгоритмы автоматической проверки программного кода (Automated Judging). Проведен сравнительный анализ систем изоляции процессов на уровне ОС (chroot, аппаратная виртуализация, контейнеризация Docker), по результатам которого Docker был выбран как наиболее безопасный и ресурсоэффективный механизм.
Проектирование Backend-архитектуры: Спроектирована распределенная архитектура на базе платформы Node.js с применением паттерна MVC/Service Layer. Данная архитектура обеспечила высокую масштабируемость бизнес-логики геймификации. Спроектирована документо-ориентированная структура NoSQL базы данных Firebase Firestore, адаптированная под высокую скорость чтения для формирования глобальных рейтингов (Leaderboards).
Создание изолированной среды (Sandbox): Успешно реализован модуль инкапсуляции пользовательского кода на языке Java. Разработанный механизм динамического создания и уничтожения Docker-контейнеров с жесткими квотами (Cgroups) гарантирует безопасность хост-сервера от всех известных векторов атак на исчерпание ресурсов (DDoS, Memory Leak, Fork Bomb) и несанкционированный доступ к файловой системе.
Интеграция с Unity-клиентом: На стороне игрового клиента разработана продвинутая сетевая подсистема. Написаны скрипты-обертки (wrappers) вокруг UnityWebRequest для авторизованного доступа по токенам (JWT). Успешно решена проблема асинхронного уведомления клиента путем создания кастомного обработчика (DownloadHandlerScript) протокола Server-Sent Events (SSE). Реализован мост сообщений (window.postMessage) для бесшовной передачи данных между веб-окружением браузера и изолированной WebGL-сборкой игры.
Тестирование: Проведено многоуровневое тестирование, подтвердившее безопасность системы проверки кода и стабильность работы потоковых уведомлений под нагрузкой.
Полученный программный комплекс (Backend + Unity Network Core) полностью удовлетворяет современным требованиям индустрии разработки образовательного ПО. Разработанное API является независимым и может быть легко масштабировано или интегрировано с мобильными приложениями и другими клиентскими интерфейсами. Дальнейшее развитие проекта предполагает внедрение микросервисной архитектуры с использованием Kubernetes для динамического масштабирования пула Judge-контейнеров при пиковых нагрузках, а также поддержку дополнительных языков программирования (Python, C++).
